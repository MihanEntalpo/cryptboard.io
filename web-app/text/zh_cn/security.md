# CryptBoard 安全性

## CryptBoard 是如何工作的？

* 数据在客户端（浏览器中）加密，也只会在客户端解密，而且只有正确的接收方才能完成解密
* 数据以加密形式经由服务器传输，服务器没有能力解密
* 客户端读取数据后，数据会立即从服务器上删除
* 客户端在服务器上的授权是匿名的，除用于防御 DoS 攻击的 IP 地址外，不需要任何用户信息
* 非对称公钥的交换通过你自行选择的第三方渠道完成

## 加密机制的详细工作方式

* 客户端以匿名方式在服务器上完成认证并获取 uid（唯一 UUID），用于发送和接收消息
* 使用 RSA + AES 混合加密算法
* 客户端会生成一对非对称 RSA 密钥（公钥 + 私钥），长度为 1024 位（为了兼顾浏览器可用性而限制密钥长度；在较慢设备上生成 2048 位密钥可能需要一分钟左右）
* 你可以通过以下第三方渠道将公钥分享给另一位客户端：
    * 即时通讯工具
    * 用手机扫描展示出的二维码
    * 通过网盘发送二维码或链接
    * 电子邮件
    * 发布到你自己的网站
* RSA 私钥**绝不会**离开你的浏览器，也不会被传输到任何地方
* 公钥不会发送到服务器，因此即使客户端侧随机数生成器质量很差，也无法利用服务器去反推出私钥
* 发送消息时，会生成一个 256 位的对称 AES 密钥，并使用它来加密数据
* AES 密钥本身会用接收方的 RSA 公钥加密，并附加到加密消息中
* 如果你有多个接收方，消息会为每个接收方分别加密
* 因为只有持有 RSA 私钥的人才能解密完整消息，而私钥不会发送到任何地方，所以除合法接收方外，没有人能解密消息
* 加密数据会连同接收方 uid 一起发送到服务器，这样服务器才知道消息应该交给谁。但服务器无法解密，因此不会知道消息内容
* 接收方从服务器取得消息后，会先用自己的 RSA 私钥解密由发送方加密的 AES 密钥，然后再用该 AES 密钥解密消息载荷

## 我为什么可以信任 CryptBoard？

* 该应用被有意设计得尽量简单，JavaScript 文件没有经过压缩或混淆，因此你可以直接审查代码
* 主应用文件 <a href='/js/frontend.js' target=_blank>frontend.js</a> 以及其他一些小文件都很容易分析。搜索 “private” 一词，就能看到所有与私钥相关的代码位置
* 未来我还计划把与密钥相关的代码集中到某个“关键区域”，让代码审计更容易
* 加解密使用了开源库，主要是 JSEncrypt 和 Forge。我本人不是专业密码学家，因此这里依赖第三方对这些库的分析（以后会补充相关论文链接）
* 所有 JavaScript 模块都取自官方仓库，可以像 <a href='/about' class="simple">About</a> 页面所述那样进行比对
* 你完全不需要信任后端服务器，因为整个项目从一开始就是按照“即便服务器被攻破也无法解密消息”的思路设计的
* 任何人都可以部署 self-hosted 的 CryptBoard 服务器，并自行控制它。

## 如何最大限度地安全使用 CryptBoard？

* 添加新的接收方公钥和 uid 时，请检查头像图像。如果你添加的是自己的另一台设备，只需看两个头像是否一致
* 如果你添加的是其他用户的 uid 和公钥，最好通过语音通话口头描述你看到的头像，并确认对方看到的是同一个。头像基于 uid 和公钥的 sha256 哈希生成，哪怕只改动一个字节，头像也会完全不同
* 你也可以直接比对 uid 和公钥本身，这样更麻烦，但更安全
* 如果你没有某个接收方的公钥且不能确定安全性，请不要向其发送消息。默认情况下这项行为是禁用的，但可以手动开启（系统会给出警告）。没有公钥的接收方会显示问号头像。**发送给此类接收方的消息不会被加密。**
* 使用完服务后，请点击顶部面板（或汉堡菜单）中的“骷髅与交叉骨”按钮，清除所有数据
* 不要在你无法信任的设备上使用本服务。如果别无选择，至少请在浏览器中使用 Incognito 模式，并在使用后清除所有数据

## 可能的安全风险以及缓解方式

* 中间人攻击（Man-In-The-Middle）：如果有人给了你一个你没有认真核对的 uid 和公钥，那么真正持有该密钥的人就可以读取你误以为发给别人的消息。因此至少要检查头像；如果你更谨慎，也应当核对 uid 和公钥本身
* 代码替换：理论上，在你审查过我的应用代码并确信其安全之后，我（或任何托管该服务的人）仍然可以秘密修改代码，使其变得不安全。解决办法是自行部署服务器
* 开源加密库中可能存在错误或后门：这种风险始终存在，所以我们只能依赖密码学分析并寄希望于最好的结果
* 由于设备熵不足等原因导致生成的 RSA 密钥较弱：未来我会加入允许用户指定自行通过 OpenSSL 生成的 RSA 密钥的功能
